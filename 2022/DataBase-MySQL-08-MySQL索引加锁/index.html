MySQL 索引加锁

MySQL 数据加锁准则：2个原则，2 个优化，1个bug

  • 2 个原则
  1. 单位为 next_key，前开后闭
  2. 访问到的对象才加锁
  • 2 个优化
  1. 唯一索引等值查询，找到等值条件时后，next-key lock 退化为行锁。
  2. 非唯一索引等值查，找到满足条件加next-key lock，然后向右遍历加锁，直到到最后一个值时退化为间隙锁
  • 1 个bug
  1. 唯一索引范围查询需要找到不满足条件的第一个值为止

不等号条件中的等值查询分析

  • 分析：
    1. order by id desc：从右往左找，先找右边界 id=12，在表 t 中，找到 (10,15]。由于是主键唯一索引，退化为间隙锁。
    2. 往左遍历进行范围查找，找到满足 id>9 的第一个左边界为 id=5，因为扫描到 id=5，会该行上加 next-key lock (0,5]。
  • 因此，该查询加锁为 (0,5]，(5,10]，(10,15)。

等值查询过程分析

等值 sql-1

  • 分析：
    1. 执行 c = 5，通过非唯一索引树找到 c=5 的行，加锁 (0,5]，向右遍历找到 c=10，满足优化2，加锁 (5,10)
    2. 执行 c = 10，加锁范围 (5,10]，(10,15)
    3. 执行 c=20，加锁范围 (15,20]，(20,25)
  • 因此，该范围的加锁顺序是 c=5, c=10, c= 20；加锁范围为：(0,5]，(5,10]，(10,15)，(15,20]，(20,25)

等值 sql-2

  • 分析：

    加锁范围和“等值sql-1”的加锁范围一致，但由于存在 order by c desc，所以两者加锁顺序不一致。sql -2 的顺序为 c=20，c=10，c=5。

    潜在问题：sql-1 和 sql-2 需要锁住相同资源，但是加锁顺序相反，因此 sql-1 和 sql-2 并发执行时，可能出现死锁。

      • “for update” 的语句持有 c=10 和 c = 20 的锁，在等 c=5 的锁；

      • 锁是一个一个加的，要尽量避免死锁，对同一组资源最好按照相同的访问顺序。
      • 发生死锁的时刻， for update 的语句占有资源更多，回滚成本更大，所以 InnoDB 选择了成本小的 lock in share mode 的语句进行回滚。

锁等待分析

  • 场景： delete 导致间隙变化：session A 加锁资源： (5,10)，(10,15]。并没有锁住 id=10 这一行，所以一开始 session B 可以删除 id=10 ，但是重新 insert id=10 就被锁了。

    session A session B
    begin;
    select * from t where id > 10 and id <= 15 for update;
    delete from t where id = 10;
    (Query OK)

    insert into t values(10,10,10);
    (blocked)

  • 通过 show engine innodb status; 查看锁等待信息。在 TRANSACTIONS 这一节

    分析：

    结果：由于 session B 执行的 delete 语句把 id=10 这一行删掉了，导致 session A 中原来两个间隙 (5,10) 和 (10,15) 变成了一个 (5,15)

update 情况下的锁分析

session A session B
begin;
select * from t where c < 5 lock in share mode;
update t set c = 1 where c = 5;
(Query OK;)

update t set c = 5 where c = 1;
(blocked)

  • 分析

    1. session A 在索引 c 上加锁范围成：为：(5,10]、(10,15]、(15,20]、(20,25] 和 (25,,supernum]

      由于 c > 5 找到的第一个记录为 c=10，因此不会给 c=5 加 (0,5] 的next-key lock

    2. session B 上第一个update 语句可以拆成 insert(c=1,id=5)； delete(c=5,id=5)。经过这个操作之后，session A的上锁范围变成：(1,10]、(10,15]、(15,20]、(20,25] 和 (25,,supernum]

    3. 执行 session B 的第二个update语句：insert(c=5,id=5)；delete(c=1,id=5)的时候，试图在 (1,10]之间 insert(c=5,id=5)，因此blocked。

小结

  • 加锁规则：2个原则2个优化1个bug
  • 通过 explain的结果，分析 SQL 与的执行流程，对索引组织表、索引、锁应该有比较清晰的了解。
  • 间隙锁取决于查找记录的间隙的两个边界，如果边界发生变化，那么间隙锁住的范围也会有所变化。
