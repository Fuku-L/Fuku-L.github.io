Spring 的事务机制

所有的数据访问技术都有事务处理机制，这些技术提供了API来开启事务、提交事务来完成数据的操作，或者在发生错误的时候回滚数据。

Spring 的事务机制是通过统一的机制来处理不同数据访问技术的事务处理。
Spring 的事务机制提供了一个 PlatformTranscationManager 接口，用来定义统一的事务处理机制。不同的数据访问技术的事务使用不同的接口实现。

数据访问技术 实现
JDBC DataSourceTransactionManager
JPA JpaTransactionManager
Hibernate HibernateTransactionManager
JDO JdoTransactionManager
分布式事务 JtaTransactionManager

1. 声明式事务

Spring 支持通过注解来选择使用事务的方法，使用 @Transactional 注解表明某个 public 方法需要事务的支持，这是一个基于 AOP 的实现操作。

2. @Transactional的事务行为

  1. propagation: 定义了事务的生命周期
  2. isolation： 定义了事务的完整性或者是事务的隔离级别
  3. timeout：定义事务过期时间，默认为当前数据库的过期时间
  4. readOnly：指定当前事务是否是“只读事务”
  5. rollbackFor：指定哪个或者哪些异常可以引起事务的回滚 (项目 highlight_spring_transaction 中在 serviceImpl 中演示了使用。)
  6. noRollbackFor：指定哪个或者那些异常不需要引起事务的回滚

2.1 事务的生命周期propagation

  1. REQUIRED[必须的]：方法 A 调用时没有事务就新建一个事务，在 A 中调用另一个方法 B 时，方法 B 使用相同的事务；如果方法 B 发生异常需要数据回滚，整个事务数据回滚。
  2. REQUIRES_NEW[需要新的]：对于方法 A 和 B，在方法调用时无论是否有事务都会开启一个新的事务，即：方法 B 中有异常，不会导致方法 A 中的数据回滚。
  3. NESTED[嵌套]：和 REQUIRES_NEW 类似，支持 JDBC，不支持 JPS 和 Hibernate。
  4. SUPPORTS[支持事务]：方法调用时有事务就用事务，没有就不用事务。
  5. NOT_SUPPORTED[不支持事务]：强制方法不在事务中执行，若有事务，在方法调用到结束阶段事务都将被挂起。
  6. NEVER[从不]：强制方法不在事务中执行，若有事务则抛出异常。
  7. MANDATORY[强制性的]：强制方法在事务中执行，若无事务则抛出异常。

2.2 事务的完整性/隔离级别isolation

  1. READ_UNCOMMITTED[读未提交]：A 事务里修改了一条记录当没有提交事务，在 B 事务中可以读到修改后的记录。可导致脏读、不可重复读以及幻读。
  2. READ_COMMITTED[读已提交]：在 A 事务中修改了一条记录且提交事务之后，B 事务才可以读取到提交后的记录。阻止脏读，可导致不可重复读以及幻读。
  3. REPEATABLE_READ[可重复读]：当 A 事务读取了一条记录，B 事务可以读取该条记录，但不允许修改这条记录。可阻止脏读和不可重复读，可出现幻读。
  4. SERIALIZABLE[序列化读]：所有事务都是顺序执行的。避免避免脏读、不可重复读和幻读。
  5. DEFAULT[默认]：使用当前数据库的默认隔离级别
